GPS Playback limitations

We are using the GPS tracks recorded with a Buell motorcycle around Regensburg in Germany. The tracks were recorded with a Garmin GPS device and transformed into GPX format without losing any information. They are real and therefore realistic!

Track Visualization

You can analyze the tracks with any GIS software, for example with QGIS, and visualize them on a map.
Let’s have a look at the track RGB-BUELL-NE-070922a.gpx and its visualization in QGIS:

In the left image you can see the trackpoints of the track, which are (artifically!) connected by straight lines. The motorcycle is driving from top to bottom, i.e. from south to north. In this special case the GPS signals are also distorted by hills and dense forests.
It is obvious that the track cannot serve as a realistic track for a Driver Assistence System.
For a Navigation System it is not a problem, since it does not actually steer the vehicle, but only provides navigation instructions. We do need the road network with a certain precision to snap the track points to the road network. In the middle image you can see the track with arrows and a small "rounding" algorithm applied to the track points.

Another way to visualize the track is using OpenStreetMap Tools - the image on the right. If we have a look at the curves of the track shown above, we can see a high precision of track points, which are actually in the middle of the road. This is not only a result of numerous track points, but the roads are actually modeled against various real world data sources.

Note that the Map projection is different in the right image, which is helpful for editing.

In the following image you can see four tracks with trackpoints in different colors
recorded on the same route in different directions:

Obviously the playback of all four tracks would be a nightmare for a Driver Assistence System. Programmed collisions and wrong speed and acceleration values would be the result of the distorted GPS signals. Nevertheless all trackpoints together roughly outline the real road and we can tell the different driving sides of the road in different directions.

Since the ROAF is all about realistic simulations we could use the road data for guidance. Be aware that road data only describes the geometry of the road, but does not contain any information about the traffic, weather, grip. And it does not contain vehicle information to actually drive the vehicle. And another challenging aspect is the GPS traffic to transfer many track points in real time.

For the time being we can agree that no one can drive this road segment at speed of 100 km/h.
Strictly due to the physical laws of motion, inertia and mass!

Track Recording

As described in the Integrated Tracker Development we still need to create features for the gps-tracker:

  • ❏ time / distance intervals

  • ❏ angular change (course over ground)

  • ❏ motion start/stop

These features are actually related to the recording of the track, which is a crucial aspect for the realism of the track. Trackers can be configured to record tracks with different settings, which can lead to very different tracks. Trackers for different purposes and with different hardware and software components can produce very different tracks. The highest precision of track recording is achieved with a high recording frequency and additional sources like a Differential GPS (DGPS) or a Real Time Kinematic (RTK) GPS, which can provide centimeter-level accuracy.

We will not implement these features in the GPS Playback development environment GpsTrackerIT !

The reason becomes apparent when we look at the track points in the images above. Just imagine we would apply time / distance intervals and / or angular changes. There would be a massive regression in the track quality and the track would be even more unrealistic than it already is.

Vice versa, if you record a track with your vehicle you should apply small angular changes to trigger the recording exactly at the turning points. This way the recorded track polyline becomes more realistic and the route length comes closer to the real curvy route length.

gps Timer

After we have the messages in the right format, we need to take care of the timing of the sending from a gps-tracker. We are using the GpsPlayerIT as out dev environment similar to the GpsTrackerIT .

As a starting point we define the Player API with the following method:

      public synchronized boolean playOsmAndTrack() {

Note that the method is synchronized to ensure thread safety when multiple trackers are involved. And be aware that the player is already loaded with the track messages from the parser. We can define the method signature and call it from the test, while the implementation of the timer will be added step by step. The method will be responsible for scheduling the sending of messages based on their timestamps.

The actual player implementation is similar to the parser design with Camel
with (1) a route invoking (2) a bean method:

(1) We are using a PlayerTimerRoute to send messages to the timer component

public class PlayerTimerRoute extends RouteBuilder {

  @Override
  public void configure() {
    from("quartz://" + ...
            :
            // room for additional parameters (trigger, job store, misfire handling, .. )
            :
            .bean("quartzSchedulerBean", "sendCurrentAndScheduleNext");

with the Consumer Endpoint URI "quartz://" + . . . for the Camel Quartz component
which eventually will trigger the sending of messages to the Producer Endpoint

(2) the bean method sendCurrentAndScheduleNext in the QuartzSchedulerBean class.

This player design is the separation of concerns between the timer route and
the scheduling logic in the bean, which will finally schedule and send the messages:

player.tracker.sendMessage(MessageOsmand message);

Geo Helpers

As the framework grows the supporting libraries and helper classes grow as well. During player development we faced the new constellation of two consecutive track points and the need to calculate speed and bearing between them.

This has brought up the TrackPoint record in our GeoTools class and
the following helper methods to calculate speed and bearing between two track points:

         speedMetersPerSecond(TrackPoint previous, TrackPoint current)
     normalizedBearingDegrees(TrackPoint previous, TrackPoint current)
     private helper normalizeBearing(double bearing)

The GpxToMessageOsmandParser was updated to use

    speed   = GeoTools.speedMetersPerSecond(previous, point);
    bearing = GeoTools.normalizedBearingDegrees(previous, point);

we’ll take care of the units [m/s] to [km/h] to [mph] etc. , when we get there.

Outlook

The main idea of a gps-player is to re/play the GPS track of a single device
by feeding each position update into a gps-tracker component.

The real/istic GPS Track can be recorded with a real GPS device, tranformed into any file format with any GIS, OpenStreetMap or geo software in general - even online. Besides using files we can also access a database (like on the Traccar server) to extract position updates.

real world traces

GPS traces are more than just a list of positions at certain times. If they are recorded with a real vehicle with a hardware GPS device they actually contain a lot more information about the environment and the vehicle itself.

Motorcycle race tracks are a good example. The team analyzes the GPS traces of the motorcycle to optimize the performance of the bike and the rider - against the track and down to track segments - on that very day with given grip, weather and so on.

The ROAF has one very central constraint: everything is live!
In order to actually simulate a RealVehicle we need to collect a lot of parameters as we drive.

On the other hand the RealTimeController should grow to a mature ROAF validator (see ROAF TT) against reality.
In case of vehicles we are talking about classical physics of motion.

synthetic tracks

Of course we can also create synthetic tracks with a track editor or even generate them with a script. The problem is the definition of the timestamps. These actually should describe the physics of the movement and are crucial for a realistic simulation.

The challenge of the ROAF will be to apply a RealTimeController to validate the real-time behavior of the system.

map segments

A digital map describes a route in terms of links and edges, which is nothing but a sequence of GPS Positions connected by straight lines. A navigation system is able to extract a path from current location to a destination Position. Yet this path or route does not provide timestamps and is purely spatial.

Player > Navigator > Driver

We can extract routes from a digital map and then tell our Device, or rather the vehicle carying it to follow this route with adopted speed. This way the gps-player can become a (proven) starting point for a Navigation System.

Once we will be able to navigate a Device, we can even go a step further:
We can implement some self driving algorithms to create a Driver and when we add environmental information from the RealTimeController we can create something like an ADAS, a Driver Assistence System.
