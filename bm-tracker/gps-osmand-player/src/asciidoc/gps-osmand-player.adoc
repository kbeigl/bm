= The GPS Player

We have prototyped the `gps-tracker` as a `@Component` that we can 'attach' to any thing. +
The  
https://github.com/kbeigl/jeets/tree/master/jeets-clients/jeets-player[`jeets-player`]
software has turned out to be a very helpful tool at development time. +
Without it every developer would need a hardware tracker which requires complex handling + 
or even worse, someone to drive around with it – during working hours. +
With a player software you can chose of thousands of available tracks and replay them. + 
Beyond that a number of simulations can be used to replay complex traffic situations.

=== Track Recording

As described in the
xref:../../../../bm-traccar/traccar-realtime-client/src/asciidoc/integrated-dev-cycle.adoc#_integrated_tracker_development[Integrated Tracker Development]
we still need to create features for the `gps-tracker`:

- [ ] time / distance intervals
- [ ] angular change (course over ground)
- [ ] motion start/stop

These features are actually related to the recording of the track, which is a crucial aspect for the realism of the track.
Trackers can be configured to record tracks with different settings, which can lead to very different tracks. 
Trackers for different purposes and with different hardware and software components can produce very different tracks. 
The highest precision of track recording is achieved with a high recording frequency and additional sources like a
Differential GPS (DGPS) or a Real Time Kinematic (RTK) GPS, which can provide centimeter-level accuracy. 

_We will *not* implement these features in the GPS Playback development environment_  `GpsTrackerIT` ! 

The reason becomes apparent when we look at the track points in the images above.
Just imagine we would apply time / distance intervals and / or angular changes.
There would be a massive regression in the track quality and the track would be even more unrealistic than it already is.

Vice versa, if you record a track with your vehicle you should apply small angular changes 
to trigger the recording exactly at the turning points. 
This way the recorded track polyline becomes more realistic 
and the route length comes closer to the real curvy route length.

== gps Timer

After we have the messages in the right format, we need to take care 
of the timing of the sending from a `gps-tracker`.
We are using the  `GpsPlayerIT` as out dev environment similar to the `GpsTrackerIT` .

As a starting point we define the Player API with the following method:

[source,java]
----
      public synchronized boolean playOsmAndTrack() {
----

Note that the method is synchronized to ensure thread safety when multiple trackers are involved. 
And be aware that the player is already loaded with the track messages from the parser.
We can define the method signature and call it from the test, 
while the implementation of the timer will be added step by step. 
The method will be responsible for scheduling the sending of messages based on their timestamps.

The actual player implementation is similar to the parser design with Camel +
with *(1)* a route invoking *(2)* a bean method:

*(1)* We are using a  `PlayerTimerRoute` to send messages to the timer component

[source,java]
----
public class PlayerTimerRoute extends RouteBuilder {

  @Override
  public void configure() {
    from("quartz://" + ...
            :
            // room for additional parameters (trigger, job store, misfire handling, .. ) 
            :
            .bean("quartzSchedulerBean", "sendCurrentAndScheduleNext");
----

with the Consumer Endpoint URI `"quartz://" + . . .` for the Camel Quartz component +
which eventually will trigger the sending of messages to the Producer Endpoint 

*(2)* the bean method `sendCurrentAndScheduleNext` in the `QuartzSchedulerBean` class.

This player design is the separation of concerns between the timer route and +
the scheduling logic in the bean, which will finally schedule and send the messages:

        player.tracker.sendMessage(MessageOsmand message);

== Geo Helpers

As the framework grows the supporting libraries and helper classes grow as well.
During player development we faced the new constellation of two consecutive track points 
and the need to calculate speed and bearing between them. 

This has brought up the TrackPoint record in our `GeoTools` class and + 
the following helper methods to calculate speed and bearing between two track points:

[source,java]
----
         speedMetersPerSecond(TrackPoint previous, TrackPoint current)
     normalizedBearingDegrees(TrackPoint previous, TrackPoint current)
     private helper normalizeBearing(double bearing)
----

The `GpxToMessageOsmandParser` was updated to use 

[source,java]
----
    speed   = GeoTools.speedMetersPerSecond(previous, point);
    bearing = GeoTools.normalizedBearingDegrees(previous, point);
----

we'll take care of the units `[m/s]` to `[km/h]` to `[mph]` etc. , when we get there.


== Outlook

The main idea of a `gps-player` is to re/play the GPS track of a single device +
by feeding each position update into a `gps-tracker` component.

The real/istic GPS Track can be recorded with a real GPS device,
tranformed into any file format with any GIS, OpenStreetMap or geo software in general - even online.
Besides using files we can also access a database (like on the Traccar server) to extract position updates.


=== real world traces

GPS traces are more than just a list of positions at certain times.
If they are recorded with a real vehicle with a hardware GPS device
they actually contain a lot more information about the environment and the vehicle itself. 

Motorcycle race tracks are a good example.
The team analyzes the GPS traces of the motorcycle to optimize the performance of the bike and the rider
- against the track and down to track segments - on that very day with given grip, weather and so on.

The ROAF has one very central constraint: everything is live! +
In order to actually simulate a `RealVehicle` we need to collect a lot of parameters as we drive.

On the other hand the `RealTimeController` should grow to a mature ROAF validator (see ROAF TT) against reality. +
In case of vehicles we are talking about classical physics of motion.


=== synthetic tracks

Of course we can also create synthetic tracks with a track editor or even generate them with a script.
The problem is the definition of the timestamps.
These actually _should_ describe the physics of the movement and are crucial for a realistic simulation.

The challenge of the ROAF will be to apply a `RealTimeController` to validate the real-time behavior of the system. 


=== map segments

A digital map describes a route in terms of links and edges, which is nothing but
a sequence of GPS Positions connected by straight lines.
A navigation system is able to extract a path from current location to a destination Position.
Yet this path or route does not provide timestamps and is purely spatial.


=== Player > Navigator > Driver

We can extract routes from a digital map and then tell our Device, or rather the vehicle carying it 
to follow this route with adopted speed. 
This way the `gps-player` can become a (proven) starting point for a Navigation System.

Once we will be able to navigate a Device, we can even go a step further: +
We can implement some self driving algorithms to create a `Driver`
and when we add environmental information from the `RealTimeController` 
we can create something like an ADAS, a Driver Assistence System.
